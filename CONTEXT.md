# Albion Black Market Data Context

This context describes the terms used for collecting and publishing the Black Market data used by Albion Blackmarket Reader.

## Language

**跨重啟保留**:
Collector 重啟後仍可載入本機保存的目前訂單狀態；只有資料年齡超過一小時時，才失去即時計算資格。

**黑市資料範圍**:
Collector 只保存地點為 Black Market 的訂單；城市市場與其他地點不屬於本機擷取資料，繼續由既有 API 提供。

**區域標記**:
黑市資料必須標記為 US、EU 或 Asia；優先由遊戲伺服器自動判斷，無法可靠判斷時要求使用者手動指定，不猜測區域。

**擷取全量**:
Collector 保存所有成功解析且屬於黑市範圍的訂單；網站支援物品、品質納入範圍與新鮮度等限制在計算階段套用。

**有效買單**:
具備正數量、正單價、可判讀且尚未到期的黑市買單，並且其擷取資料仍在一小時新鮮度內；無法確認有效性的訂單只保留追蹤，不參與價格計算。

**目前訂單狀態**:
每個黑市訂單只保留其最新已知狀態，不代表完整觀測歷史；同一訂單的新觀測會更新舊狀態，歷史成交統計仍由 API 提供。

**品質納入範圍**:
五種品質的訂單都保留；Normal、Good、Outstanding 參與第一版黑市價格與利潤計算，Excellent、Masterpiece 暫不納入計算但仍可被追蹤。

**本機即時資料**:
資料從遊戲流量被擷取後，到本機 Dashboard 顯示的資料；目標延遲上限為 5 秒。GitHub 與 Vercel 的發布延遲不屬於本機即時資料的延遲。

**資料新鮮度**:
本機擷取資料從擷取時間起算的可接受年齡；黑市資料超過 1 小時未更新時，不再視為可用的即時資料。

**黑市快照**:
某一時間點、特定區域與地點可觀察到的黑市訂單資料集合；它描述當下可見的市場狀態，不等同於歷史成交統計。

**純被動監聽**:
只觀察遊戲正常產生並送達本機的流量，不修改、注入、重送封包，也不代替玩家操作遊戲。

**線上發布快照**:
由 GitHub commit 建立並部署到 Vercel 的黑市資料版本；它不是本機即時資料，更新時間包含 push、部署與快取延遲。

**手動發布**:
只有使用者明確執行發布動作時，才把本機黑市快照提交到 GitHub；本機擷取器不會自動 commit 或 push。

**資料來源**:
一筆黑市資料的取得途徑與時間，包括本機擷取或 Albion Data API；來源與年齡應能被使用者辨識。

**即時優先回退**:
對每個物品、區域與地點的黑市資料，有本機擷取結果時優先採用；該筆本機資料沒有可用結果時，才採用既有 Albion Data API 的資料。部分物品擷取成功時，其他物品仍可各自回退到 API。

**最佳有效買單價**:
在同一物品、區域與地點的本機黑市快照中，所有尚未過期且仍有數量的買單之最高單位價格。

**訂單品質**:
黑市訂單本身標示的品質層級；它會隨訂單保存，即使第一版物品清單仍以未拆分品質的方式呈現。

**本機訂單 JSON**:
本機保存黑市目前訂單狀態的 JSON 檔案；擷取器以訂單 ID 更新資料，本機 Dashboard 直接讀取它，手動發布時由此狀態產生網站發布投影。

**逐項發布合併**:
手動發布時，只有具備一小時內有效本機資料的物品、區域與地點才覆蓋既有發布資料；沒有本機資料的項目保留原本由 Albion Data API 產生的資料。

**訂單消失處理**:
訂單未出現在後續觀測時，不立即刪除其最後狀態；在遊戲回傳的到期時間已過，或最後觀測時間超過一小時後，才不再納入計算，舊資料可再清理。

**本機整合入口**:
本機封包擷取器、JSON 資料與 React Dashboard 透過既有的 .NET `localhost:5173` 入口整合；Vercel 仍只提供手動發布後的線上快照。

**封包擷取驅動**:
第一版使用 Npcap 這類一次性安裝的被動封包擷取元件；擷取器只讀取遊戲正常產生的流量，不修改、注入或重送封包。驅動未安裝或權限不足時，資料流程回退到既有 Albion Data API。

**品質內部保留、外部扁平化**:
本機 JSON 保留五種品質與訂單細節；計算黑市 `bm` 時只取 Normal、Good、Outstanding 的最高有效買單價，發布仍使用既有 `{ id, bm, sold }` 格式，`sold` 繼續來自 API 歷史資料。

**發布投影流程**:
手動發布以目前 Git 中的 `bm-crafter-{region}.json` 作為 API fallback 基底，只覆蓋一小時內有效的本機項目並寫回既有格式；發布指令不重新呼叫 API，也不自動 commit 或 push。

**區域發布防呆**:
已知遊戲伺服器 IP 時自動標記 US、EU 或 Asia；無法判斷時必須手動指定，且自動判斷與手動指定衝突時停止寫入，避免把資料發布到錯誤區域。

**逐筆來源與時間**:
發布資料每筆 item 可附 `source`（`local` 或 `api`）與 `observedAt`；既有 `id`、`bm`、`sold` 欄位維持不變。UI 可依來源篩選，並顯示該筆資料時間；缺少 metadata 的舊資料預設為 API，時間使用檔案的 `generatedAt`。

**解析失敗回退**:
無法解析或不完整的封包直接跳過，不寫入不確定的訂單；該項目繼續使用 Albion Data API。Collector 顯示最近成功擷取時間與解析錯誤數，原始封包除非手動開啟除錯模式，否則不保存。

**手動發布介面**:
第一版透過 CLI 或 PowerShell 指令產生發布 JSON，不在 Dashboard 提供發布按鈕；指令只寫檔並輸出變更摘要，Git commit 與 push 仍由使用者另行手動執行。

**本機狀態與發布檔分離**:
本機專用 JSON 保存完整黑市訂單狀態，作為擷取器的真實狀態；Git/Vercel 使用既有 `bm-crafter-{region}.json` 保存網站需要的發布投影，兩者用途與格式分離。

**JSON 損壞復原**:
本機 JSON 每次先寫入暫存檔並驗證，再以原子替換更新；同時保留上一版 `.bak`。主檔損壞時使用備份，主檔與備份都無法讀取時回退到 Albion Data API。

**本機頁面更新**:
本機 Dashboard 不自動輪詢或推送更新；Collector 持續寫入本機 JSON，使用者透過瀏覽器手動重新整理頁面取得最新資料。瀏覽器重新整理只重新讀取本機資料與既有 API 快照，不觸發 API pipeline。
